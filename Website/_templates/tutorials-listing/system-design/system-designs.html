üìå Introduction to System Design

System Design is the process of defining the architecture, modules, interfaces, data flow, and components for a system to satisfy specified requirements. It bridges the gap between problem statements and code implementation.

System design is crucial for building scalable, maintainable, and performant software systems. Whether you're designing a small feature or an enterprise-scale application, you must consider both high-level and low-level aspects.

üèóÔ∏è 1. What is High-Level Design (HLD)?

‚úÖ Definition:

High-Level Design (HLD) represents the blueprint of the system. It focuses on architecture, external interfaces, and major modules. HLD is about what the system does and how components interact with each other ‚Äî not their internal implementation.

üîç Goals:

  • Define the overall system architecture

  • Identify major components/services/modules

  • Choose technology stack

  • Plan communication flow between modules

  • Ensure scalability, availability, and fault tolerance

üìê Common Outputs:

  • System Architecture Diagrams

  • Component Diagrams

  • Data Flow Diagrams (DFD)

  • Deployment Diagrams

  • Basic Database Design (Entity names only)

  • Tech Stack Choices

  • Load balancer, CDN, Cache usage overview

üì¶ Example (Instagram - HLD):

  • Services: User Service, Feed Service, Media Service, Comment Service

  • Data Storage: PostgreSQL (users/posts), Redis (caching), Amazon S3 (images)

  • Communication: REST APIs, Kafka for event-based flow

  • External: CDN for fast media delivery

üîß 2. What is Low-Level Design (LLD)?

‚úÖ Definition:

Low-Level Design (LLD) focuses on the detailed internal implementation of individual modules/components. It answers how each module works, how data is processed, and how the components integrate at the code level.

üîç Goals:

  • Define classes, functions, and interfaces

  • Create detailed ER diagrams and database schema

  • Design API endpoints, request/response formats

  • Choose appropriate design patterns

  • Ensure code-level scalability and modularity

üìê Common Outputs:

  • Class Diagrams

  • ER Diagrams (with relations, indexes, constraints)

  • API Contracts with request/response formats

  • Sequence Diagrams

  • Flowcharts or Pseudocode

  • Folder/module structure

üß± LLD Checklist:

Element Details
Classes User, Post, Comment, Notification, etc.
Functions createPost(), getFeed(), uploadImage(), etc.

API Design REST/GraphQL with validation, authentication, throttling
Design Patterns Singleton, Factory, Adapter, Strategy, Repository
Error Handling Try-catch blocks, failover strategies
Testing Unit tests, mocks, integration tests

üöÄ Real-World Example ‚Äì Designing a Scalable URL Shortener (Like Bitly)

‚úÖ High-Level Design:

  • Services:

    • Frontend Service: UI to input long URL and display short one

    • URL Service: Generates and stores shortened URLs

    • Redirect Service: Maps short URL to original

  • DB: Use NoSQL for speed (e.g., DynamoDB), or SQL with indexes

  • Cache: Redis to cache frequently accessed URLs

  • Load balancer and horizontal scaling

  • CDN for frontend files

  • Analytics service for tracking

‚úÖ Low-Level Design:

  • Class ShortURL: id, original_url, short_code, created_at

  • API: POST /shorten, GET /{short_code}

  • Logic: Base62 encoding for short codes

  • Table schema:

    sql
    CREATE TABLE urls (
  id SERIAL PRIMARY KEY,
  short_code VARCHAR(10) UNIQUE,
  original_url TEXT NOT NULL,
  created_at TIMESTAMP DEFAULT now()
);

  • Retry logic for code collisions

  • Tests for invalid URLs

üß† Common Mistakes to Avoid

  1. Skipping HLD ‚Äì Jumping straight into code without planning architecture.

  2. Overengineering LLD ‚Äì Too many classes/functions for simple tasks.

  3. Ignoring scalability/failure points in HLD.

  4. Not updating HLD after scaling decisions (e.g., caching added later).

  5. Hardcoding business logic in LLD ‚Äì hurts modularity and testing.
